# 06. Non-Functional Requirements

## Introduction

Non-functional requirements (NFRs) specify the operational criteria, quality standards, and system behaviors that define how the **Generative AI-Powered Cloud Security Assistant** operates. These ensure the system is secure, performant, resilient, usable, and compliant with enterprise software standards.

---

## 1. Performance and Latency

### Log Processing Speed
* **NFR-1.1**: The ingestion and normalization module must process a single cloud log payload (up to 100 KB) in under **200 milliseconds** from ingestion to queue writing.

### AI Reasoning Output Latency
* **NFR-1.2**: For interactive chat workflows, token streaming must initiate within **1.5 seconds** of prompt submission.
* **NFR-1.3**: The complete threat analysis generation, RAG compilation, and formatting must compile in under **10 seconds** under average load conditions.

### Vector Search Performance
* **NFR-1.4**: Top-k similarity retrieval queries to the Vector Database (with up to 10,000 documents) must resolve in under **100 milliseconds**.

---

## 2. Security and Privacy (Data Protection)

### Cryptographic Protections
* **NFR-2.1**: All network communication between the user's browser, the application gateway, databases, and external LLM APIs must be encrypted using **TLS 1.3** protocols.
* **NFR-2.2**: Sensitive databases (incident history, configuration credentials, metadata) must employ **AES-256** encryption at rest.

### Token & Credential Shielding
* **NFR-2.3**: Under no circumstances should raw cloud credentials (API tokens, IAM credentials, access keys) found in security logs be transmitted to external LLM API endpoints.
* **NFR-2.4**: The system must run a deterministic pre-processor that strips all character patterns matching standard AWS access patterns (`AKIA[A-Z0-9]{16}`), Base64 patterns, and email formats.

### Application Security Compliance
* **NFR-2.5**: The web interface and APIs must be hardened against the **OWASP Top 10** vulnerabilities (including Cross-Site Scripting, SQL Injection, and Broken Object Level Authorization).

---

## 3. Reliability and Availability

### Operational Uptime
* **NFR-3.1**: The system dashboard and processing gateway must target **99.9% availability** (three nines), excluding planned maintenance windows.

### Fault Tolerance & API Downtime Degradation
* **NFR-3.2**: If the external LLM provider API goes offline or encounters rate-limiting (HTTP 429), the assistant must gracefully downgrade:
  1. Notify the user of API unavailability.
  2. Fall back to local, rule-based matching schemas.
  3. Load relevant remediation guidelines directly from the local vector database cache.
* **NFR-3.3**: The integration layer must use **exponential backoff and retry** logic to handle transient remote network drops.

---

## 4. Scalability and Portability

### Horizontal Autoscaling
* **NFR-4.1**: The backend services (FastAPI/Node.js) must be fully containerized (Docker) to enable horizontal scaling behind a load balancer during peak security alerts.
* **NFR-4.2**: The queue system must buffer spikes of up to **10,000 events per minute** without dropping log events or causing system crashes.

### Cloud Integration Independence
* **NFR-4.3**: The system architecture must be cloud-agnostic, allowing components to run on AWS (EKS), Microsoft Azure (AKS), Google Cloud (GKE), or local server resources.

---

## 5. Usability and Accessibility

### User Experience Design
* **NFR-5.1**: The user interface must support responsive scaling, presenting clear structures across standard desktop viewports (1920x1080 to 1366x768).
* **NFR-5.2**: The interface must feature a dark mode toggle to reduce cognitive strain for SOC analysts working night shifts.
* **NFR-5.3**: Commands and code blocks generated by the LLM must be rendered inside markdown syntax structures featuring a one-click "Copy to Clipboard" button.

### Confidence Indicators
* **NFR-5.4**: The AI engine must display a clear assessment indicator showing the relevance and source matching score of the retrieved RAG contexts (e.g., "Source match confidence: 92%").

---

## 6. Compliance and Policy Alignment

### GDPR / Data Privacy Compliance
* **NFR-6.1**: The database must support "Right to Be Forgotten" logic, allowing administrators to permanently delete specific logs containing user names or IP histories.

### SOC2 / ISO 27001 Alignment
* **NFR-6.2**: The system must log all user requests, system configurations, and AI outputs to a write-once-read-many (WORM) audit database collection to maintain compliance trails.
