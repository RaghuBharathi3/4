# 10. Modular Architecture

## Module breakdown

The **Generative AI-Powered Cloud Security Assistant** is composed of seven decoupled functional modules. Below is a detailed analysis of each module's purpose, operational interfaces, boundaries, and complexities.

---

## 1. Log Ingestion & Parsing Module

### Operational Specifications
* **Purpose**: Ingests, normalizes, and sanitizes multi-cloud JSON security log streams.
* **Inputs**: Raw JSON alert payloads via API Webhooks (AWS GuardDuty, CloudTrail, Azure Defender, GCP SCC).
* **Outputs**: Sanitized, normalized telemetry events matching the internal standard event model schema.
* **Responsibilities**:
  * Exposes highly available API ingress points.
  * Validates webhook signatures and origin headers.
  * Implements queuing (Redis queue) to buffer traffic spikes.
  * Runs the PII Redaction pipeline (regex + NER) to remove private data.
* **Future Scalability**: Can be refactored into cloud-native serverless functions (e.g., AWS Lambda, Azure Functions) triggerable directly by event brokers.
* **Challenges**: Managing large event bursts during security incidents without losing packets or memory exhaustion.
* **Dependencies**: FastAPI, Redis, Pydantic.
* **Estimated Complexity**: Low to Medium (2/5).

---

## 2. Threat Detection & Context Enrichment Module

### Operational Specifications
* **Purpose**: Gathers structural and intellectual security context corresponding to incoming alerts.
* **Inputs**: Normalized event objects.
* **Outputs**: Enriched alert context (relevant CIS guidelines, MITRE techniques, and matching CVE details).
* **Responsibilities**:
  * Vectorizes query parameters using the embedding model API.
  * Queries the Vector Database to pull top-k similar safety standards chunks.
  * Matches event characteristics to specific threat descriptors.
* **Future Scalability**: Connects to dynamic threat feeds (e.g., TAXII/STIX protocols) to auto-update local intelligence indexes.
* **Challenges**: Vector search latency during high concurrent alert volumes.
* **Dependencies**: FAISS/ChromaDB database driver, Embedding API client.
* **Estimated Complexity**: Medium (3/5).

---

## 3. AI Reasoning & Threat Explanation Module

### Operational Specifications
* **Purpose**: Coordinates cognitive interpretation of incidents using LLMs.
* **Inputs**: Normalized events + Enriched RAG context + System prompt configurations.
* **Outputs**: Explanatory incident narratives, vulnerability walkthroughs, and remediation steps.
* **Responsibilities**:
  * Assembles prompt variables.
  * Manages connection sessions to LLM endpoints.
  * Handles token-streaming pipelines to broadcast real-time tokens to user channels.
* **Future Scalability**: Supports dynamic routing between models (e.g., using small local models for low-severity logs and calling large models for critical threats).
* **Challenges**: Managing token-limit overheads when dealing with highly nested cloud log files.
* **Dependencies**: LangChain/LlamaIndex API wrappers, target LLM client SDKs.
* **Estimated Complexity**: Medium-High (4/5).

---

## 4. Remediation & Action Engine Module

### Operational Specifications
* **Purpose**: Builds valid mitigation command configurations and Infrastructure as Code adjustments.
* **Inputs**: Structured AI outputs (remediation suggestions).
* **Outputs**: Interactive code blocks containing vendor-specific CLI syntaxes and Terraform templates.
* **Responsibilities**:
  * Filters and isolates executable commands inside secure dashboard display components.
  * Prevents automatic backend script execution (maintaining the human-in-the-loop gate).
* **Future Scalability**: Can integrate with ticket-management systems (Jira, ServiceNow) to auto-scaffold remediation tickets.
* **Challenges**: Ensuring CLI options generated by the LLM don't contain deprecated vendor flags.
* **Dependencies**: Python standard library template parsers, PyYAML.
* **Estimated Complexity**: Medium (3/5).

---

## 5. Security Reporting & Metrics Module

### Operational Specifications
* **Purpose**: Aggregates incident telemetry to compile analytical compliance summaries.
* **Inputs**: Persistent database query records, timeline tables.
* **Outputs**: High-fidelity PDF/Markdown compliance summaries and posture analysis tables.
* **Responsibilities**:
  * Compiles timelines of security alerts.
  * Renders markdown elements into standard page-numbered document designs.
  * Generates metric dashboards summarizing MTTR, MTTD, and alert-severity volumes.
* **Future Scalability**: Scheduled email distribution to executive leadership and compliance auditors.
* **Challenges**: Compiling complex multi-step timelines without performance bottlenecks on database lookups.
* **Dependencies**: ReportLab (Python PDF library) or Puppeteer-based HTML-to-PDF generators.
* **Estimated Complexity**: Low-Medium (2.5/5).

---

## 6. Admin Query & Chat Interface Module

### Operational Specifications
* **Purpose**: Powers conversational interactions and historical semantic lookup for admins.
* **Inputs**: Natural language inquiries, session tokens.
* **Outputs**: Contextual chat answers referencing database archives.
* **Responsibilities**:
  * Maintains conversational history states (session tokens).
  * Executes vector lookups against old database events.
  * Streamlines the UI state between chat windows and system alert panels.
* **Future Scalability**: Adding voice-activated security query capabilities.
* **Challenges**: Preserving state across distributed chat sessions without memory leaks.
* **Dependencies**: WebSockets libraries, JSON schemas.
* **Estimated Complexity**: Medium (3/5).

---

## 7. Security Guardrails & Compliance Auditing Module

### Operational Specifications
* **Purpose**: Secures system operations, prevents prompt injection, and audits output safety.
* **Inputs**: Raw prompts + Raw LLM response texts.
* **Outputs**: Audited strings (passes inspection or halts processing).
* **Responsibilities**:
  * Scans inputs for known injection strategies (e.g. system-override hacks).
  * Evaluates LLM script generation output syntax to prevent credential leakages.
  * Log all actions to the secure system audit DB.
* **Future Scalability**: Integration of real-time machine learning-based exploit scanners at the API level.
* **Challenges**: Keeping inspection filters lightweight to prevent processing bottlenecks.
* **Dependencies**: Guardrails AI or NeMo Guardrails, regex engines.
* **Estimated Complexity**: High (4.5/5).
